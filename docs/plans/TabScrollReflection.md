# 标签页切换滚动偏移问题深度回顾

## 1. 问题背景
在多标签页编辑器中，用户在不同文件之间切换时，滚动位置无法精准恢复，表现为：
- **内容向上或向下漂移**：恢复后的视觉焦点与离开时有几行误差。
- **直接跳到文档底部**：滚动位置完全丢失，停留在文档末尾。
- **固定位置跳转**：无论离开时在哪，切回来总是强制对齐到某一行的行首。

## 2. 核心技术根源
该问题的复杂性在于它是多个底层机制共同作用的结果：

### A. CodeMirror 6 的虚拟化渲染 (Virtualization)
CodeMirror 6 为了性能，只渲染视口内的 DOM。视图外的文档内容高度是基于文本行数的**估算值**。
- **后果**：当你尝试恢复一个像素级的 `scrollTop` 时，如果编辑器尚未对上方未渲染区域进行精确测量，它的总高度可能与保存时不同，导致同一个像素值指向了不同的逻辑位置。

### B. 异步组件与装饰器的“高度欺骗”
- **图片解码延迟**：即便图片在本地，浏览器在解码完成前无法提供宽高，导致初始高度为 0。
- **语法高亮与装饰器**：代码块、大标题等在着色或添加 CSS 类后高度会发生微跳变。
- **布局塌陷**：CodeMirror 在 [setState](file:///e:/require/xiao-a-note/src/modules/editor/components/CodeMirrorEditor.tsx#433-500) 的瞬间会测量文档高度。如果在这一微秒内图片或代码块尚未完成布局，测量结果会瞬间“塌陷”，导致滚动被浏览器强制截断（Clamp）到底部。

## 3. 失败方案回顾

### 方案一：绝对像素恢复 (`scrollTop`)
- **实施**：直接记录并恢复 `view.scrollDOM.scrollTop`。
- **失败原因**：像素值依赖于不稳定的 `scrollHeight`。在虚拟滚动环境下，文档总高度随手滚动动态变化，绝对值无法作为唯一锚点。

### 方案二：基于 `scrollIntoView` 的行定位
- **实施**：记录视口顶部行号，切回时调用 `scrollIntoView(line, { y: 'start' })`。
- **失败原因**：`y: 'start'` 机制过于生硬，会将目标行强制贴齐顶边，导致用户丢失了行内的精细滚动位置（例如用户正看在某一行中间时，切回来变到了行首）。

### 方案三：精准锚点系统 (行号 + 像素偏移)
- **实施**：同时记录 `topLineNumber` 和该行距离视口顶部的 `topOffset`。
- **失败原因**：
    - **时序竞争**：保存操作 (`REQUEST_SAVE_CURSOR`) 与 React 组件销毁、文档切换之间存在毫秒级的时序误差。
    - **事件断路**：在某些版本中，NoteEditor 的事件监听逻辑未被正确挂载，导致保存了空的或旧的数据。
    - **坐标系不稳定**：CodeMirror 在 [resetState](file:///e:/require/xiao-a-note/src/modules/editor/components/CodeMirrorEditor.tsx#433-500) 后的一两帧内，其内部坐标系仍在频繁变动，单纯的 `requestAnimationFrame` 无法捕捉到“最终稳定态”。

### 方案四：三段式防御 (行锚点 + 多阶梯定时器 + 图片监听)
- **实施**：记录 `topLineNumber` 和 `topOffset`，并结合 `requestAnimationFrame` 和 `setTimeout` 进行多轮修正，同时监听全容器的 `load` 事件。
- **失败原因**：
    - **保存点的“虚假测量”**：在 React 准备销毁旧文件组件的瞬间，`getScrollState` 捕获到的 `scrollTop` 可能是 0 或异常值，因为此时 DOM 结构可能已经开始坍塌。
    - **视口抖动 (Jitter)**：多轮定时器的逐级修正虽然能最终对齐，但在用户眼中表现为内容“跳动”多次，体验极差。
    - **锚点行失效**：在极端复杂的文档（如嵌套极其深的代码块或未设置高度的 Widget）中，`lineBlockAtHeight` 在未完整测量前返回的坐标系与恢复后的坐标系存在像素级偏差。

## 4. 总结与反思
滚动恢复看似简单，实则是**浏览器渲染流水线**、**CodeMirror 布局引擎**与**React 组件生命周期**之间的同步博弈。我们目前的“被动补偿”方案（即出问题后再跳回来）已经走到了死胡同。未来的修复需要考虑“主动锁定”：
1. **预占位机制**：对图片和 Widget 强制设置高度占位，消除布局跳变。
2. **同步保存机制**：在 React 卸载动作发生**之前**，抢先一步完成状态锁定。
3. **坐标系持久化**：研究是否可以将 CodeMirror 内部的 `ViewState` 部分持久化，而不仅仅是简单的行号。
